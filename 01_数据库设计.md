# 设计一:数据库设计

> AI 角色世界 · 唯一真相源(Single Source of Truth)
> 引擎设计见 `02_引擎程序设计.md`,提示词契约见 `03_提示词与模型交互设计.md`。

## 0. 总原则

1. **数据库是世界的唯一真相**。任何数值,只有写进库里才算数;LLM 在对话里说的任何数字都不作数。
2. **属性分两类**:
   - **写死(static)** = 这个人/这个世界"是谁",定义风格,基本不变。
   - **可改(dynamic)** = 当前状态,定义走向,随事件流变。
3. **存储选型**:SQLite 单文件,开 `PRAGMA foreign_keys=ON`。复杂结构用 TEXT 存 JSON(SQLite 的 JSON1 函数够用)。时间用整数 `tick`(世界时钟)做主轴。
4. **事件流是底座**:`scene_log` 是 append-only 的,世界的一切发生都先落这张表 —— 它既是回放源,也是"成长/变化"的原始素材。

---

## 1. 时间模型

一个整数 `tick` 是世界唯一时钟。派生关系:

```
1 tick   = 一个最小行动单元(一句话 / 一个 action)
1 day    = N_TICKS_PER_DAY 个 tick(配置,如 24)
1 chapter= N_DAYS_PER_CHAPTER 天(配置,如 7)—— "章"是成长结算与世界演化的大周期
```

`chapter`(章)对应 Agentopia 里的"年":章末做角色 profile 漂移、世界线推进、关系沉淀。这是"成长"和"世界变化"被结算的节点。

## 2. 角色层

### 2.1 characters —— 写死的人格内核

```sql
CREATE TABLE characters (
  id            TEXT PRIMARY KEY,          -- 'char_lin' 等稳定 ID
  name          TEXT NOT NULL,
  age           INTEGER,
  gender        TEXT,
  occupation    TEXT,                      -- 初始职业/社会角色
  backstory     TEXT NOT NULL,             -- 背景故事
  personality   TEXT NOT NULL,             -- JSON 0-100: {外向,攻击性,诚实,谨慎,共情,自制力}
  talents       TEXT NOT NULL,             -- JSON: 各技能领域的"上限" {格斗:80,口才:30}
  values        TEXT NOT NULL,             -- JSON 数组: 核心动机 ["守护妹妹","渴望认可"]
  speech_style  TEXT NOT NULL,             -- 说话风格/口癖(角色卡一致性命根子)
  taboos        TEXT,                      -- JSON 数组: 雷点触发器 ["背叛","被轻视"]
  created_tick  INTEGER DEFAULT 0
);
```

> `personality` 与 `taboos` 是裁定层最关键的输入:同一件事,自制力 20 与 80 的人,裁定结果应当不同。
> `talents` 是**上限**不是当前值,它卡住技能能成长到多高 —— 这是"成长有天花板、不会无限膨胀"的机制。
> 注意:`personality` 虽然在内核表里,但允许章末做**极缓慢漂移**(见 2.6),这是"人也会变"的成长表达。

### 2.2 char_states —— 可改的动态状态(1:1)

```sql
CREATE TABLE char_states (
  char_id       TEXT PRIMARY KEY REFERENCES characters(id),
  location_id   TEXT REFERENCES locations(id),  -- 当前在哪
  mood          INTEGER DEFAULT 50,        -- 0-100
  energy        INTEGER DEFAULT 100,
  health        INTEGER DEFAULT 100,
  intoxication  INTEGER DEFAULT 0,         -- 酒馆专属:醉酒度
  hunger        INTEGER DEFAULT 0,
  money         INTEGER DEFAULT 0,
  reputation    INTEGER DEFAULT 50,        -- 名声
  fulfillment   TEXT NOT NULL,             -- JSON: {mood,material,social,esteem} 各0-100
  current_goals TEXT,                      -- JSON 数组:短期意图,会被剧情改写
  updated_tick  INTEGER DEFAULT 0
);
```

状态每 tick 由代码自然衰减(energy↓、intoxication 随时间代谢、fulfillment 享乐适应缓降),细节在引擎文档。

### 2.3 skills —— 技能(成长核心)

```sql
CREATE TABLE skills (
  char_id  TEXT REFERENCES characters(id),
  name     TEXT NOT NULL,                  -- '口才','格斗','酿酒'
  level    INTEGER DEFAULT 0,              -- 0-100, 上限受 talents 中对应领域约束
  xp       INTEGER DEFAULT 0,              -- 经验累积,满则 level+1
  PRIMARY KEY (char_id, name)
);
```

**成长规则(代码执行,见引擎文档 §6):** 行为命中某技能 → `xp += k` → 满阈值则 `level++`,但 `level` 不得超过 `talents[领域]`。用进废退:长期不用可缓慢回落(可选)。

### 2.4 inventory —— 物品

```sql
CREATE TABLE inventory (
  id       INTEGER PRIMARY KEY AUTOINCREMENT,
  char_id  TEXT REFERENCES characters(id),
  item     TEXT NOT NULL,
  qty      INTEGER DEFAULT 1,
  props    TEXT                            -- JSON:物品属性
);
```

### 2.5 relationships —— 有向关系(每对一行,A→B 可≠ B→A)

```sql
CREATE TABLE relationships (
  from_id     TEXT REFERENCES characters(id),
  to_id       TEXT REFERENCES characters(id),
  affection   INTEGER DEFAULT 0,           -- 0-100 好感
  trust       INTEGER DEFAULT 0,
  respect     INTEGER DEFAULT 0,
  familiarity INTEGER DEFAULT 0,           -- 熟悉度(决定能否互动)
  memory_text TEXT,                        -- A 对 B 的自由文本印象(LLM 维护)
  updated_tick INTEGER DEFAULT 0,
  PRIMARY KEY (from_id, to_id)
);
```

> 有向是关键:暗恋、单方面记仇都能自然表达。`affection / respect` 两条线后续可直接跑社交评分(谁更被喜欢/尊重)。

MVP 约定：没有 `A → B` 行代表 A 尚未直接认识 B；一次直接对话或明确指向 B 的行动，会由本地规则建立 `A → B` 与 `B → A` 两行，并将 `familiarity` 设为初识值。之后的直接互动缓慢提高熟悉度；好感、信任、尊重仍必须经过裁定层。

### 2.6 personality_drift_log —— 人格漂移记录(章末成长)

```sql
CREATE TABLE personality_drift_log (
  id        INTEGER PRIMARY KEY AUTOINCREMENT,
  char_id   TEXT REFERENCES characters(id),
  chapter   INTEGER,
  trait     TEXT,                          -- 哪个人格轴
  delta     INTEGER,                       -- 变化量(限制在 ±3 这种极小范围)
  reason    TEXT,                          -- 章末总结给出的理由
  ts        INTEGER
);
```

章末由"世界模型"读这一章的经历,对个别人格轴给极小漂移(如长期被欺压 → 自制力↓、攻击性↑)。漂移**有硬上限**,防止人格崩坏。

### 2.7 memory —— 记忆(分级)

```sql
CREATE TABLE memory (
  id        INTEGER PRIMARY KEY AUTOINCREMENT,
  char_id   TEXT REFERENCES characters(id),
  kind      TEXT NOT NULL,                 -- 'episodic'(单事件) / 'summary'(章摘要) / 'note'(自管笔记)
  content   TEXT NOT NULL,
  salience  INTEGER DEFAULT 50,            -- 重要度,决定是否进上下文、是否被淘汰
  chapter   INTEGER,
  ts        INTEGER
);
CREATE INDEX idx_memory_char ON memory(char_id, salience DESC, ts DESC);
```

**记忆管理(防长跑膨胀,见引擎文档 §7):** episodic 随手记;章末把本章 episodic 压成一条 summary;低 salience 的老 episodic 定期淘汰。进上下文时只取"高 salience summary + 最近 episodic",不是全量。

## 3. 世界层

### 3.1 world_state —— 世界单例

```sql
CREATE TABLE world_state (
  id             INTEGER PRIMARY KEY CHECK (id = 1),  -- 永远只有一行
  sim_tick       INTEGER DEFAULT 0,
  day            INTEGER DEFAULT 0,
  chapter        INTEGER DEFAULT 0,
  period         TEXT DEFAULT 'morning',   -- 昼夜:morning/noon/evening/night
  weather        TEXT DEFAULT 'clear',
  season         TEXT DEFAULT 'spring',
  economy_index  INTEGER DEFAULT 50,       -- 宏观经济(影响物价/收入)
  tension        INTEGER DEFAULT 0,        -- 紧张度,门控冲突类事件
  active_threads TEXT,                     -- JSON:进行中的剧情线 [{id,desc,stage,involved}]
  worldview      TEXT NOT NULL             -- 世界观长文(写死,世界模型的根 context)
);
```

`tension / economy_index / active_threads` 是"世界有变化"的载体:它们随事件累积而变,又反过来门控下一批事件。`active_threads` 让多 tick 的长剧情(如"债主追讨")能跨场景延续。

### 3.2 locations —— 地点(也有状态)

```sql
CREATE TABLE locations (
  id          TEXT PRIMARY KEY,            -- 'hall'(大厅),'kitchen','room_2','street'
  name        TEXT NOT NULL,
  description TEXT,
  is_public   INTEGER DEFAULT 1,           -- 公共/私密
  state       TEXT,                        -- JSON:动态状态 {整洁度,人气,损坏}
  connected   TEXT                         -- JSON 数组:可达地点 ['hall','street']
);
```

地点有状态(被斗殴砸坏、打烊关门)→ 世界会"留下痕迹",而非每场重置。

### 3.3 events —— 事件(定义 + 实例)

```sql
CREATE TABLE event_defs (                  -- 事件模板(设计期写死)
  id         TEXT PRIMARY KEY,
  tier       TEXT NOT NULL,                -- 'scheduled' / 'weighted' / 'emergent'
  condition  TEXT NOT NULL,                -- 代码可判的触发条件表达式(见引擎 §5)
  cooldown   INTEGER DEFAULT 0,            -- 触发后冷却 tick 数
  weight     INTEGER DEFAULT 10,           -- weighted 层的基础权重
  guidance   TEXT NOT NULL                 -- 给世界模型的生成引导(世界观片段+本事件意图)
);

CREATE TABLE event_instances (             -- 实际触发的事件
  id           INTEGER PRIMARY KEY AUTOINCREMENT,
  def_id       TEXT REFERENCES event_defs(id),
  tier         TEXT,
  fired_tick   INTEGER,
  location_id  TEXT,
  narration    TEXT,                       -- 世界模型生成的事件描述
  payload      TEXT,                       -- JSON:环境 deltas / 影响范围
  visibility   TEXT,                       -- JSON:谁能感知 ['char_lin','char_a'] 或 'all'
  status       TEXT DEFAULT 'active'       -- active/resolved
);
```

三层 tier 的设计意图(条件示例在引擎 §5):
- **scheduled**:到点必触发(打烊、换日)。
- **weighted**:从池子按权重抽,但权重被 world_state 门控(下雨才解锁"屋顶漏水")。
- **emergent**:状态跨阈值才触发(`affection>85 且 自制力<40` → 醉后表白)。**最出戏的一层,重点投入。**

## 4. 事件流 / 日志(回放与素材底座)

### 4.1 scene_log —— append-only 世界事件流

```sql
CREATE TABLE scene_log (
  id          INTEGER PRIMARY KEY AUTOINCREMENT,
  tick        INTEGER NOT NULL,
  scene_id    INTEGER,                     -- 归属场景
  actor_id    TEXT,                        -- 谁(角色 / 'WORLD' / 'JUDGE')
  type        TEXT NOT NULL,               -- 'speech'/'action'/'event'/'verdict'/'narration'
  content     TEXT NOT NULL,               -- 台词或叙事
  data        TEXT,                        -- JSON:结构化附加(action 详情/裁定 deltas)
  visibility  TEXT                         -- 可见范围
);
CREATE INDEX idx_scenelog_scene ON scene_log(scene_id, tick);
```

> 这是整个系统的脊柱:**模拟只管往这张表 append,前端从这张表回放**。模拟与观看彻底解耦(引擎 §8)。成长结算、章末摘要也都从这张表读素材。

### 4.2 scenes —— 场景生命周期

```sql
CREATE TABLE scenes (
  id           INTEGER PRIMARY KEY AUTOINCREMENT,
  location_id  TEXT REFERENCES locations(id),
  start_tick   INTEGER,
  end_tick     INTEGER,
  participants TEXT,                       -- JSON:在场角色
  turn_budget  INTEGER,                    -- 回合预算(防死循环)
  status       TEXT DEFAULT 'open'         -- open/closed
);
```

## 5. 配置表

```sql
CREATE TABLE config (
  key   TEXT PRIMARY KEY,
  value TEXT
);
-- 例:N_TICKS_PER_DAY=24, N_DAYS_PER_CHAPTER=7, FULFILLMENT_DECAY=2,
--     SCENE_DEFAULT_BUDGET=12, STALE_THRESHOLD=3, SUMMARY_EVERY_TICKS=40
```

把所有可调参数集中此处,引擎运行时读取,方便不动代码调平衡。

## 6. 字段白名单(executor 闸门用)

裁定层产出的 delta 只允许改下列字段,executor 据此校验后才落账(引擎 §4):

| target | 允许的 field | 约束 |
|---|---|---|
| char_state | mood, energy, health, intoxication, hunger, money, reputation, fulfillment.* | 0-100(money/reputation 可超范围按配置) |
| skill | xp | 只增,level 由代码推导,≤ talent 上限 |
| rel | affection, trust, respect, familiarity, memory_text | 0-100 |
| inventory | item(add/remove) | qty ≥ 0 |
| world | tension, economy_index | 0-100 |

> **不在白名单的字段,任何 LLM 输出都被拒。** 这一张表就堵死了数值幻觉。

### 当前 MVP 落地范围

当前引擎的内部 target 名为 `character_attribute` 与 `relationship`。角色属性先开放
`mood / energy / health / money`；关系先开放 `affection / trust / respect / familiarity`。
关系 delta 必须带 `source` 与 `ref`，只允许本拍行动者指向其直接互动的在场者；executor
负责按 0-100 夹紧并自动创建该有向关系行。其余白名单字段保留为后续扩展接口。

## 7. ER 关系速览

```
characters ─1:1─ char_states ──── locations
    │  1:N           │
    ├── skills       └── (current location)
    ├── inventory
    ├── memory
    └── relationships(自引用,有向)

world_state(单例) ── active_threads
locations ──1:N── scenes ──1:N── scene_log
event_defs ──1:N── event_instances ── (注入 scene_log)
```

建表顺序:`characters → locations → world_state → char_states → skills/inventory/relationships/memory → event_defs → scenes → event_instances → scene_log → config`。
