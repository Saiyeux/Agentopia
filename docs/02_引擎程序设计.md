# 设计二:引擎程序设计

> 数据库见 `01_数据库设计.md`,提示词契约见 `03_提示词与模型交互设计.md`。
> 技术栈:Python orchestrator + SQLite(唯一真相源)+ LM Studio OpenAI-compatible endpoint(单模型、串行)。

## 0. 设计心法

世界不是"一群 agent 在聊天",而是**一个时钟在走,每拍代码先把能算的都算完,只在需要语义判断处才掏出 LLM**。
因为 30B 本地每次调用都慢,**减少 LLM 调用次数 = 第一性能杠杆**。

每拍最少 LLM 调用:`1 次角色发言`,只有当发言含 action/有数值后果时才 `+1 次裁定`,只有触发涌现事件时才 `+1 次世界生成`。**纯寒暄要能跳过裁定层。**

---

## 1. 模块划分

```
engine/
├── clock.py        时钟:tick 推进、昼夜/章切换、状态自然衰减
├── eventsys.py     世界层:三层事件判定 + 调世界模型生成事件
├── scheduler.py    场景调度:开/关场景、选下一个发言者、冷场检测
├── agent.py        角色层:组装上下文 → 调角色模型 → 产出行动提议
├── judge.py        裁定层:代码预检 → 调裁定模型 → 产出 JSON 判决
├── executor.py     落账:校验 deltas(白名单+边界)→ 写 SQLite
├── memory.py       记忆:写入、章末压缩、淘汰、上下文检索
├── growth.py       成长:技能 xp 结算、章末人格漂移、关系沉淀
├── llm.py          LLM 客户端:调用、JSON 解析、重试、降级
├── db.py           数据访问层(DAL):所有读写经此,带校验闸门
└── loop.py         主循环:把上面串起来,while True
```

层与层只通过 **DB(真相源)** 和 **结构化消息** 通信,不互相直接改对方内存。

## 2. 主循环(自走的核心)

世界"一直跑"的本质:**结束的只是场景,世界永不结束**。每拍都保证有事可做 —— 没场景就开场景,没人行动就让世界注入事件。

```python
def run():
    while True:                                  # 世界永不停,只有场景生灭
        w = db.world()                           # 读世界单例
        clock.tick(w)                            # ① 时钟+1,衰减状态,判昼夜/换日/换章
        eventsys.scan_and_fire(w)                # ② 三层事件判定→可能注入事件

        scene = scheduler.current_or_open(w)     # ③ 取活动场景,没有就开一个
        if scene is None:                        #    实在没人可凑(深夜空馆)
            clock.fast_forward(w)                #    快进到下一个有意义的时间点
            continue

        actor = scheduler.next_speaker(scene)    # ④ 选谁行动(接力优先,导演兜底)
        proposal = agent.act(actor, scene, w)    # ⑤ 角色 LLM:产出一句话/一个action
        log(scene, actor, proposal)              #    先落 scene_log(append-only)

        if proposal.has_effect():                # ⑥ 有数值后果才进裁定
            verdict = judge.adjudicate(proposal, scene, w)   # 裁定 LLM → JSON判决
            executor.apply(verdict)              # ⑦ 校验+落账(纯代码)
            log(scene, 'JUDGE', verdict)
            growth.accrue(actor, verdict)        # ⑧ 技能xp/关系增量随裁定结算

        memory.record(actor, scene, proposal, verdict)       # ⑨ 写记忆

        scheduler.update_scene(scene)            # ⑩ 回合预算-1、冷场检测→可能关场景
        periodic_jobs(w)                         # ⑪ 周期任务:摘要/章末结算
        throttle()                               # 可选:控速,或全速跑由回放端限速
```

> `while True` 就是"自动一直跑"。停不下来的不是 bug,是设计目标。需要可控时给一个外部 `stop_flag` / 跑满 N 章自动存档退出。

## 3. 角色层(agent.act)

产出**提议**,不碰数据库。三层上下文组装(细节见提示词文档 §2):

```python
def act(actor, scene, world):
    ctx = {
      "roleplay": build_roleplay_prompt(actor),     # 写死persona + 高salience记忆摘要 + 风格/雷点
      "stage":    build_stage_prompt(scene, world),  # 当前场景:在场者、最近几拍、地点、世界事件
      "history":  scene.recent_messages(),           # 本场景多拍消息(含可见性过滤)
    }
    raw = llm.call(MODEL, system=ctx["roleplay"]+ctx["stage"], messages=ctx["history"],
                   schema=ACTION_SCHEMA)             # 要求结构化输出
    return parse_action(raw)                          # → {speech, action?, target?, visibility}
```

**关键:** 上下文里给的所有数值(钱、好感)都来自 DB 实时查询,不让角色"记账"。角色只负责"想做什么",不负责"算后果"。

## 4. 裁定层 + executor(AI 提议,代码裁决)

### 4.1 两道闸门夹一次 LLM

```python
def adjudicate(proposal, scene, world):
    # 闸门①:硬规则预检(纯代码,不调LLM)—— 不合法直接驳回,省一次调用
    ok, reason = precheck(proposal)        # 在场? 钱够? 冷却好? 目标存在?
    if not ok:
        return Verdict(success=False, narration=reason, deltas=[])

    # AI 裁定:只给"变化量delta"和叙事,绝不给绝对值,绝不碰库
    raw = llm.call(MODEL, system=JUDGE_PROMPT,
                   user=render(proposal, actor_state, present, world),
                   schema=VERDICT_SCHEMA)
    return parse_verdict(raw)              # → {success, narration, deltas[], triggers[]}
```

### 4.2 executor —— 唯一能改库的地方,纯代码

```python
def apply(verdict):
    for d in verdict.deltas:
        if d.field not in WHITELIST[d.target]:   # 白名单(数据库 §6)
            continue                              # 不在白名单 → 丢弃
        if d.op == "add":
            newval = clamp(current(d) + d.value, 0, 100)   # 边界夹紧
        ...
        db.write(d.target, d.field, newval)       # 落账
        if d.memory_append:
            db.append_rel_memory(d.from, d.to, d.memory_append)
```

> **纪律:数值变更只此一路。** 裁定模型给的是"建议的 delta",executor 用代码校验+夹边界后才 commit。LLM 永远碰不到 `db.write`。这就根除了数值幻觉与"把钱改成一百万"。

## 5. 世界层(eventsys.scan_and_fire)

三层 tier 顺序判定,条件全部**代码可判**:

```python
def scan_and_fire(world):
    for ev in event_defs():
        if on_cooldown(ev): continue
        if   ev.tier == "scheduled":   hit = eval_time(ev.condition, world)
        elif ev.tier == "weighted":    hit = roll_weighted(ev, gated_by=world)  # 状态门控权重
        elif ev.tier == "emergent":    hit = eval_state(ev.condition, world)    # 跨阈值
        if hit:
            fire(ev, world)
```

条件表达式示例(存在 `event_defs.condition`,引擎用安全求值器解析):

```
scheduled : period == 'night'                          # 打烊
weighted  : weather == 'rain'                           # 解锁"屋顶漏水",权重再随机
emergent  : rel(A,B).affection > 85 and A.自制力 < 40   # 醉后表白
emergent  : char.money < 0                              # 债主上门
emergent  : world.tension > 70                          # 一触即发的斗殴
```

`fire()` 里:**条件命中由代码判,事件的"具体演绎"交世界模型生成**(提示词文档 §4)→ 产出 narration + 环境 deltas + 可见范围 → 写 `event_instances` 并注入 `scene_log` → 在场角色下一拍上下文带上它 → 影响言行。事件造成的 `tension↑`、地点损坏等经 executor 落账,形成"世界留痕"。

## 6. 成长(growth)—— "角色有成长"的落点

成长不是新机制,是**已有数据随时间累积的自然结果**,分三个时间尺度:

| 尺度 | 触发点 | 内容 |
|---|---|---|
| 每拍 | 裁定后 `accrue()` | 命中技能 → `xp += k`,满阈值 `level++`(≤ talent 上限);关系增量随裁定 delta 落账 |
| 每章 | `periodic_jobs` 章末 | 世界模型读本章 scene_log → 给个别人格轴**极小漂移**(±1~3,写 `personality_drift_log`);把本章 episodic 压成一条 summary |
| 长期 | 关系沉淀 | familiarity 决定能否互动;affection/respect 长期累积改变社交格局 |

> 成长**有天花板**(talent 上限 + 漂移硬上限),所以长跑不会让角色数值/人格崩坏 —— 这是"简单可行"的关键约束。

---

## 7. 长跑生存(防漂移、防膨胀)—— "一直跑"的工程保障

30B 长跑必然漂移、复读、OOC、上下文爆炸。四道防线:

1. **周期 re-grounding**:每 `SUMMARY_EVERY_TICKS` 拍,把本场景历史压成摘要替换原始多拍消息,并把 persona+当前状态硬重新注入。
2. **记忆分级检索**:进上下文只取"高 salience summary + 最近 N 条 episodic",绝不全量(数据库 §2.7)。
3. **记忆淘汰**:章末淘汰低 salience 的老 episodic,只留 summary,防表无限增长。
4. **response filtering**:角色输出过两道 —— 代码查实体幻觉(提到注册表外的人/地点 → 驳回重生成),裁定/世界模型查是否 OOC、是否违反角色雷点(提示词文档 §5)。

## 8. 模拟与回放解耦 —— "简单可行"的观感保障

现实:30B 上每拍 1~3 次调用、~45 tok/s,一个 50 拍的"酒馆之夜"要十几分钟。**不要让自己盯进度条。**

```
[后台进程] 全速模拟 ──append──▶ scene_log(SQLite)
                                     │
[前端/Kotlin-Compose] ◀──poll/tail── 以"可读速度"回放,像看一部自动生成的剧
```

模拟端只管把世界往前推、往 `scene_log` 写;观察端独立读这张表回放。两者完全异步,互不阻塞。你已有的 group-chat 原型可直接当回放器。

## 9. 调度细节(scheduler)

### 9.1 选发言者:接力优先,导演兜底(省 LLM 调用)

```python
def next_speaker(scene):
    last = scene.last_action()
    if last and last.target:              # A 明确对 B 说话 → 直接交给 B,零额外调用
        return last.target
    if is_stale(scene):                   # 冷场(连续 STALE_THRESHOLD 拍无新信息)
        return None                       # 触发关场景或注入事件
    if scene_open(scene):                 # 开放场景没人被点名 → 调一次轻量"导演"
        return director_pick(scene)       # 唯一会多花 LLM 的分支
    return round_robin(scene)             # 退化:轮询
```

### 9.2 场景生命周期

- 进入 → 若干回合(受 `turn_budget` 限制)→ 退出。
- 角色可主动 `exit`(action 之一)离场。
- `turn_budget` 耗尽 / 冷场 → 代码强制 `close_scene()`,把"在场者散场后的去向"交还给下一拍调度。
- 关场景时可触发一条"场景小结"事件,世界 tension/active_threads 据此更新。

## 10. 容错(llm.py)

```python
def call(model, ..., schema, retries=2):
    for i in range(retries+1):
        raw = openai_call(...)               # LM Studio endpoint
        obj = try_parse_json(raw)            # 容忍 ```json 包裹、尾随文本
        if valid(obj, schema): return obj
        ...                                   # 重试时在 user 末尾追加"上次输出非法,请严格按schema"
    return fallback(schema)                   # 仍失败 → 返回安全默认(如 success=false,空deltas)
```

裁定/世界模型输出非法时,**宁可让这一拍"无事发生"也不让坏数据落库**。executor 那道闸是最后保险。

## 11. 启动流程

```
1. init_db()                        建表(数据库文档顺序)
2. seed_world()                     载入 worldview、locations、event_defs
3. seed_characters()                离线生成的 8~12 张角色卡灌入 characters/relationships/char_states
4. config 写入运行参数
5. run()                            while True
```

角色与世界的**初始生成是一次性离线任务**,可用云端强模型做(质量更高、不占本地算力),只把结果灌库。运行期只用本地 30B。
