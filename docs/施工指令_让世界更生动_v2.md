# Agentopia 施工指令：让世界更生动 v2（去偏执 + 去单调）

> 给 Claude Code 的本地施工文档。本版基于已同步的最新代码（含 threading system、drama 旋钮）重写，before/after 对应真实实现。
> 目标：(1) **偏执**——所有角色围着同一条剧情线程反复处理、岔不开；(2) **单调**——永远同一拨人在同一地点轮流说话。

---

## 诊断（已对照真实代码，务必先读）

**偏执的真正病根在 `app/threads.py` 的机制里，不只是提示词。** 三处叠加：

1. **线程永不因停滞退场。** `_next_stage` 只能单向前进，线程唯一出路是在 `executing` 阶段被 `_contains_resolution` 命中"完成/搞定/查到了"等关键词。**没有超时、没有搁置、没有 parked 状态**——没人恰好说出收束词，线程就永远 `active`。
2. **收束判定脆弱且自相矛盾。** `_contains_resolution` / `_contains_commitment` 都被 `_looks_like_question` 一票否决：句子带"吗/么/怎么/多少"就既不算承诺也不算收束。线程极易卡在 `executing` 出不来。
3. **`thread_situation` 每拍强推 threads[0]。** 它把最高优先级的那条未收束线程，用强制口吻塞给当前 actor（"本拍要承接它…普通追问、报价、检查、准备动作都不能当作结局"）。配合前两条，这条线就成了所有人每拍被按头处理的东西。

**单调是结构性的：** scheduler 在单一 scene 内 round-robin 全体在场角色——一个地点、一桌人、不移动、不分组。提示词救不了，必须靠多场景（阶段 C）。

**优先级修正：阶段 B（机制）现在与阶段 A（提示词）同等重要，甚至更优先。** 提示词只能缓解，机制才是解药。

> 注：第一份文档的 drama 旋钮 / target_need 已落地，本文件不重复。

## 阶段 A：提示词再平衡（先做，零代码风险）

编辑 `backend/prompts/*.md`，保存。以下 before/after 对应磁盘现有原文。

### A1. `character_system.md`

**规则 13，改为：**
> 13. 如果 `active_story_threads` 非空，承接最相关的一条推进它。但若该线程的 `stale` 为真、或 `stage` 已停在某阶段多拍没有实质进展，就主动收束它、或暂时搁置转向别的事。搁置不是放弃。

**规则 14，改为：**
> 14. ambient world events 多为背景，但当它与你的目标/性格相关、**或当前线程已停滞**时，可以让它真正改变你这拍的关注点。世界不该对发生的事永远无动于衷。

**新增规则 16、17（去偏执关键）：**
> 16. 不必每拍都谈目标或线程。角色也会做与目标无关的小事、对别人的话题产生临时兴趣、观察、走神、闲扯。这些"无用"的细节正是世界显得活的原因。
>
> 17. 不同角色对同一件事关注度不同。**当已有人在处理某条线程时，其他人应去推进各自不同的关注点，而不是一起围观同一个问题。** 与你性格/目标无关的线程，可以只旁观、简短回应或岔开。

### A2. `character_task_instruction.md` —— 去掉"首要任务"绝对化

**整段改为：**
> 作为 actor，你这拍可以推进一条 `active_story_threads`、推进自己的 `short_term_goals`、**或只是自然地活在场景里**——按情境择一，不必每拍都服务任务。
> 若推进线程：问清条件、提出交换、分工、执行一步或汇报结果；讨论阶段不能假装完成。
> 若线程已停滞或与你无关：搁置它，转向别的人、话题或一件与目标无关的小事。
> 说话尽量为下一步铺路，但允许偶尔只是闲谈或观察。避免机械复述旧目标。

### A3. `opening_system.md` —— 修正与世界观自相矛盾的负向指令

**问题：** 世界观已是赛博朋克（事件含 netrunner_job / cyberware / corp_intel），但提示词仍写"不要固定依赖某一组赛博朋克意象"——这在跟世界自己的主题打架，且负向指令本就弱。

**删除这两行：**
> - 不要照抄世界观、地点事实或最近已用开场中的措辞
> - 不要固定依赖某一组赛博朋克意象，允许偶尔使用环境词，但每次要换观察角度

**替换为：**
> - 拥抱赛博朋克氛围，从下列意象库中**每次选一两个、且与上几次不同**的角度切入：霓虹与全息广告的明灭、合成酒精与机油的气味、义体的微光与伺服声、酸雨打在金属棚顶、数据噪声与人声混杂、安保无人机的红点、街市叫卖的余音、电缆与管线的阴影、空气里的电离味
> - 同一场景每次开场换感官通道（这次写声音，下次写气味或光线），不复述最近开场的措辞

> 换世界观时只替换这组意象库——这是"内容 vs 引擎"那条线在提示词层的体现。

## 阶段 B：线程停滞与退场机制（核心解药，改 `app/threads.py`）

让线程能因停滞而搁置、让收束不再依赖脆弱关键词、让 `thread_situation` 不再每拍强推同一条。

### B1. 增加停滞计数（数据已就绪：`story_threads` 已有 `beat_count`）
- 在 `record_thread_turn` 里，判断这拍对线程**是否有实质进展**：`new_stage != thread["stage"]` 即为有进展。
- 给 `story_threads` 增加列 `stalled_turns INTEGER NOT NULL DEFAULT 0`（在 `db.py` 的建表处加，并在迁移段 `_ensure_column` 加兜底）。
- 有进展时 `stalled_turns = 0`；无进展（stage 未变）时 `stalled_turns + 1`。在那条 `UPDATE story_threads` 里一并写入。

### B2. 增加"搁置"退场
- 在 `record_thread_turn` 计算完 `new_stage` 后：若 `stalled_turns + 1 >= PARK_AFTER`（建议常量 `PARK_AFTER = 3`）且 `new_stage != "resolved"`，则把 `status` 置为 `'parked'`。
- `parked` 线程不删除。修改 `active_thread_slice` 与 `active_thread_count` 的 WHERE 子句，使其仅取 `status = 'active'`（现已如此），从而 parked 线程自动退出"被强推"的候选——无需删数据即可让世界喘息。
- 可选：当某 actor 主动重新提及 parked 线程标题、或相关事件再次触发时，把它置回 `active`、`stalled_turns=0`。

### B3. 修复脆弱的收束判定
- `_contains_resolution` 与 `_contains_commitment` 当前被 `_looks_like_question` 全盘否决，导致线程卡死。改为：**只有当整句以疑问结尾或纯粹是提问时**才否决承诺/收束，而不是句中出现任一疑问词就否决。简单做法：把 `_looks_like_question` 的判定收紧为"句子去掉首尾空白后以 ？/? 结尾，或以'谁/哪/为什么/能不能/要不要'开头"。
- 额外保险：在 `executing` 阶段，若 `stalled_turns >= PARK_AFTER`，即使没命中收束关键词，也允许 `_next_stage` 返回 `resolved`（按"超时自然了结"处理），避免永久卡执行。

### B4. 软化 `thread_situation` 的强制口吻
- 当前它对 threads[0] 用"本拍要承接它…不能当作结局"的硬措辞。改为：当线程 `stalled_turns >= PARK_AFTER` 时，改用允许搁置的措辞（"这条线索可以推进、收个尾、或先放一放，转向别的事"）；未停滞时维持原推进措辞但去掉"必须"。
- 不要总是只用 threads[0]：当 actor 不在 threads[0] 的 participants 里时，`thread_situation` 应返回 None（让该 actor 自由行动），而不是强行套用最高优先级线程。

### B5. 把停滞信号喂进提示词（配合阶段 A）
- `active_thread_slice` 返回的每条线程对象，增加 `stalled_turns` 与 `stale`（= `stalled_turns >= PARK_AFTER`）两个字段。
- 这两个字段会随 `active_story_threads` 进入 `build_character_prompt` 的 payload，使规则 13/14 的"已停滞"有确定性依据，模型不再靠猜。

### B 阶段验收
- 在 `watch.py` 临时打印每条活动线程的 `stage / stalled_turns / status`。
- 跑 30 拍：应观察到线程停滞约 `PARK_AFTER` 拍后变 `parked` 或被超时 `resolved`，角色随后转向别的事；不应再出现一条线程被无限反复处理。
- 同一拍里不同 actor 的 `situation` 不应再千篇一律指向同一条线。

## 阶段 C：多场景 + 角色移动（根治"场景过于单一"）

> 单调的真正根源，也是最大的工。**务必在 A、B 跑顺、用观察窗确认偏执已缓解后再动**，不要并行，避免变量太多看不清因果。

骨架（届时细化为独立文档）：
1. **多 location / 多 scene**：世界存在数个地点（如 大堂 / 后巷 / 二楼包间 / 数据黑市），scene 不再唯一。
2. **`move` 行动原语**：角色 action 增加 `move`（去某地点）。`scheduler` 不再对全体在场角色轮转，而是**按角色当前 location 分组**，各场景内部各自轮转。
3. **小群对话**：同一 location 内 2-3 人自然成组，避免"一桌五人车轮战"。
4. **场景间时间对齐**：决定多场景并行推进还是交替聚焦；后者可与 drama 旋钮联动——drama 高时切到冲突最浓的场景（这正是将来"自动电影"的导演镜头选择）。

### C 触发条件
- A、B 完成，且用 `watch.py` 跑过至少 50 拍真实（接 LM Studio）运行，能具体说出"还有哪些单调"，再开 C 的详细文档。
