> 本文档分三个阶段，**严格按顺序做**。阶段 0 零风险，先做完、先用起来；阶段 1 是核心；阶段 2 故意推迟。

## 背景（先读，决定怎么改）

引擎核心循环在 `backend/app/engine.py` 的 `step_once()`：
`advance_tick → scan_and_fire(事件) → pick_next_actor → run_character_action(角色LLM) → adjudicate_action(裁决LLM) → apply_deltas → finish_turn`。

经诊断，"世界很闷、角色像漫无目的"有三个结构性原因，**都不是 bug，是设计缺口**：

1. **裁决倾向于什么都不发生。** `judge.py` 里 `temperature=0.2`、delta 数值被夹在 ±5，提示词还要求"避免每拍都改数值"。结果大量拍是"本拍无数值变化"。
2. **随机事件极少触发，且与角色无关。** `eventsys.py` 的事件 `condition` 只能引用 `world_state` 的字段（`sim_tick/day/chapter/period/weather/tension/economy_index`），**看不到任何角色、更看不到角色目标**。现有 4 个事件全是纯环境触发。
3. **目标（goals）被存了却全程惰性。** 角色目标存在 `character_attributes` 的 `goals` 字段（JSON 数组），会进角色提示词，但：提示词第 9 条叫角色"不要复述旧目标"；目标**不进裁决提示词**；事件系统**够不到目标**。于是世界对角色想要什么毫无反应。

**核心设计原则（改动必须遵守）：**
- "真实↔精彩"的旋钮（下称 `drama`）必须作用在**确定性代码层**（事件选择），**不能**靠往提示词塞"请更戏剧化"。旋钮每动一格，都要对应世界里一个可观察、可解释、可复现的变化。
- LLM 负责"判断与叙述"，确定性代码负责"流程与裁定边界"。新增能力时这条线不能糊。
- 所有数值写库都要走 `executor.py` 的 allowlist 校验，不要绕过。

## 阶段 0：观察窗（新文件，零风险，必须最先做）

**目的**：现在所有状态变化都埋在 sqlite 里，是"盲改"。先造一个能看清每拍发生什么的窗口。

**任务**：在 `backend/` 下新建文件 `watch.py`，内容如下，**逐字创建，不要改动**：

```python
"""
watch.py — Agentopia 观察窗 / trace dumper

独立脚本，不修改引擎任何逻辑。跑 N 拍，把每一拍发生的事打印成人能读的形式，
让你第一次"看见"世界每拍在干什么：事件有没有触发、角色说了什么、裁决到底改了什么。

用法（在 backend/ 目录下）：
    python watch.py 20            # 跑 20 拍
    python watch.py 20 --reset    # 先重置角色状态再跑（每次想从干净状态看时用）

依赖你本地的 LM Studio（和引擎一样）。若模型连不上，引擎会走 fallback，
脚本仍能跑完——这时你会看到大量"裁决：本拍无变化"，那本身就是有用的信号。
"""
from __future__ import annotations

import argparse
import sys
from pathlib import Path

# 让脚本能 import 到 app.*（无论从哪运行）
sys.path.insert(0, str(Path(__file__).resolve().parent))

from app.db import connect, loads, reset_character_state, seed_demo  # noqa: E402
from app.engine import step_once  # noqa: E402


GREY = "\033[90m"
BOLD = "\033[1m"
GREEN = "\033[32m"
YELLOW = "\033[33m"
CYAN = "\033[36m"
DIM = "\033[2m"
RESET = "\033[0m"


def _name_map(conn) -> dict[str, str]:
    return {r["id"]: r["name"] for r in conn.execute("SELECT id, name FROM characters")}


def _char_state(conn, char_id: str) -> dict:
    rows = conn.execute(
        "SELECT attr_id, value FROM character_attributes WHERE char_id = ?",
        (char_id,),
    ).fetchall()
    return {r["attr_id"]: r["value"] for r in rows}


def _world(conn) -> dict:
    row = conn.execute(
        "SELECT sim_tick, day, period, weather, tension, economy_index FROM world_state WHERE id = 1"
    ).fetchone()
    return dict(row) if row else {}


def print_header(conn) -> None:
    names = _name_map(conn)
    w = _world(conn)
    print(f"\n{BOLD}=== 世界初始状态 ==={RESET}")
    print(f"  tick={w.get('sim_tick')} 天={w.get('day')} 时段={w.get('period')} "
          f"天气={w.get('weather')} 紧张度={w.get('tension')} 经济={w.get('economy_index')}")
    print(f"{BOLD}=== 角色（钱/心情/精力/健康 + 目标）==={RESET}")
    for cid, name in names.items():
        s = _char_state(conn, cid)
        goals = loads(s.get("goals"), [])
        print(f"  {name:6} 钱{s.get('money'):>3} 心情{s.get('mood'):>3} "
              f"精力{s.get('energy'):>3} 健康{s.get('health'):>3}  "
              f"{DIM}目标:{goals}{RESET}")
    print()


def print_tick(conn, result: dict, names: dict[str, str]) -> None:
    tick = result["tick"]
    events = result.get("events", [])
    actor_id = result["actor_id"]
    actor_name = result.get("actor_name", actor_id)
    action = result.get("action", {})
    verdict = result.get("verdict", {})

    if events:
        ev = events[-1]
        ev_line = f"{YELLOW}★ 事件[{ev.get('tier')}] {ev.get('title')}：{ev.get('narration')}{RESET}"
    else:
        ev_line = f"{GREY}（无事件）{RESET}"

    s = _char_state(conn, actor_id)
    goals = loads(s.get("goals"), [])

    print(f"{BOLD}[t={tick:>3}]{RESET} {ev_line}")
    print(f"   {CYAN}{actor_name}{RESET} "
          f"{DIM}(钱{s.get('money')} 心情{s.get('mood')} 精力{s.get('energy')} 健康{s.get('health')} "
          f"目标{goals}){RESET}")

    speech = action.get("speech", "")
    inner = action.get("inner", "")
    to = action.get("to")
    act = action.get("action")
    to_str = f" → {names.get(to, to)}" if to else ""
    print(f"   说{to_str}：「{speech}」" + (f"  {DIM}（心声:{inner}）{RESET}" if inner else ""))
    if act:
        print(f"   {DIM}意图: {act.get('type')} target={act.get('target')} {act.get('detail','')}{RESET}")

    applied = verdict.get("applied", [])
    changed = [a for a in applied
               if a.get("status") == "applied" and a.get("old") != a.get("new")]
    ok = "成功" if verdict.get("success") else "未成"
    if changed:
        parts = []
        for a in changed:
            owner = names.get(a.get("ref"), a.get("ref"))
            parts.append(f"{owner}.{a.get('field')} {a.get('old')}→{a.get('new')}")
        print(f"   {GREEN}裁决({ok})：{'，'.join(parts)}{RESET}")
    else:
        print(f"   {GREY}裁决({ok})：本拍无数值变化{RESET}")
    print()


def main() -> None:
    parser = argparse.ArgumentParser(description="Agentopia 观察窗")
    parser.add_argument("ticks", type=int, nargs="?", default=10, help="要跑的拍数")
    parser.add_argument("--reset", action="store_true", help="先重置角色状态再跑")
    args = parser.parse_args()

    seed_demo()
    if args.reset:
        reset_character_state()
        print(f"{DIM}（已重置角色状态）{RESET}")

    conn = connect()
    names = _name_map(conn)
    print_header(conn)

    empty_verdicts = 0
    event_ticks = 0
    for _ in range(args.ticks):
        result = step_once(conn)
        conn.commit()
        print_tick(conn, result, names)
        if result.get("events"):
            event_ticks += 1
        applied = result.get("verdict", {}).get("applied", [])
        if not any(a.get("status") == "applied" and a.get("old") != a.get("new") for a in applied):
            empty_verdicts += 1

    n = args.ticks
    print(f"{BOLD}=== 统计 ==={RESET}")
    print(f"  有事件的拍: {event_ticks}/{n} ({event_ticks*100//max(n,1)}%)")
    print(f"  无数值变化的拍: {empty_verdicts}/{n} ({empty_verdicts*100//max(n,1)}%)")
    print(f"  {DIM}→ 这两个数字越接近'事件少、空裁决多'，世界就越闷。"
          f"做导演层的目标，就是把它们往'事件多且指向角色目标、裁决常推进目标'的方向拉。{RESET}\n")

    conn.close()


if __name__ == "__main__":
    main()
```

**验收标准**：
- 在 `backend/` 下、且本地 LM Studio 已启动时，运行 `python watch.py 20` 能打印 20 拍可读 trace，末尾有统计。
- 即使 LM Studio 未启动也不能崩溃（引擎有 fallback）。
- **做完后人工跑一次 `python watch.py 20`，把"有事件的拍 %"和"无数值变化的拍 %"两个数字记下来。这是阶段 1 的基线。**

## 阶段 1：导演层旋钮（drama）—— 让事件指向角色目标

**目的**：把"真实↔精彩"做成一个数值旋钮 `drama (0–100)`，作用在事件选择这一确定性层。`drama=0` 退化成现在的纯随机忠实模拟；`drama` 越高，越倾向于触发"针对某个在场角色当前匮乏/目标"的事件。

### 1.1 给世界加一个 drama 字段
- 在 `world_state` 表增加列 `drama INTEGER NOT NULL DEFAULT 30`（在 `app/db.py` 的 `SCHEMA` 中世界表定义处添加；并确保 `init_db` 对已存在的库做一次幂等的 `ALTER TABLE ... ADD COLUMN` 兜底，避免旧库报错）。
- 取值范围 0–100。30 作为默认中档。

### 1.2 给事件加"针对的匮乏类型"标签
- 在 `event_defs` 表增加列 `target_need TEXT`（可空）。取值约定为：`money_low` / `mood_low` / `health_low` / `energy_low` / `null`。
- `null` 表示纯环境事件（现有 4 个事件保持 `null`，行为不变）。
- 在种子数据里**新增 3–4 个"压力/机会"事件**，每个带一个 `target_need`，`tier='weighted'`，并给中等 `weight`（如 40）。示例（narration/guidance 用酒馆世界观，自行润色）：
  - `target_need='money_low'`：有人进来招短工 / 债主在门外张望（指向"缺钱"的角色）
  - `target_need='mood_low'`：老熟人捎来一句宽心话 / 角落传来一段熟悉的曲子
  - `target_need='health_low'`：药师注意到某人脸色不对 / 一碗热汤被推到面前
  - `target_need='energy_low'`：后厨喊人帮忙（消耗精力但给钱）/ 有人让出靠墙的座位
- 这些事件的 `condition` 仍只用 `world_state` 字段（沿用现有 DSL，不要扩展 condition 语言）。"是否针对当前在场角色"的判断放在 Python 里做（见 1.3），**不要塞进 condition 字符串**。

### 1.3 改造 `eventsys.scan_and_fire` —— 核心改动
在 `app/eventsys.py`：
- `_world_context()` 增加读取 `drama`。
- 新增一个纯函数 `_present_needs(conn, scene) -> set[str]`：读取在场角色（`scene["participants"]`）的属性，按阈值判断匮乏，返回命中的 need 标签集合。阈值建议：`money < 20 → money_low`，`mood < 35 → mood_low`，`health < 60 → health_low`，`energy < 40 → energy_low`。
- 修改命中逻辑：对 `tier='weighted'` 的事件，计算**有效权重** `effective_weight`：
  - 若 `target_need is None`（环境事件）：`effective_weight = weight`（不受 drama 影响，保持忠实）。
  - 若 `target_need` 命中了 `_present_needs`：`effective_weight = weight + drama * K`（建议 `K=0.8`，并夹到 0–100）。即 drama 越高，针对性事件越容易触发。
  - 若 `target_need` 不为空但**未**命中在场匮乏：`effective_weight = max(0, weight - 30)`（无的放矢的针对性事件被压低）。
  - 用 `effective_weight` 替换原来 `_weighted_hit(row["weight"])` 里的 `weight`。
- `scheduled` / `emergent` 两个 tier 的行为**完全不变**（只动 weighted 这一层）。
- 不改变"每次最多 fire 的逻辑结构"，只改权重计算。

### 1.4 让观察窗能显示 drama 与 need（小改 watch.py）
- 在 `watch.py` 的 `_world()` 查询里加上 `drama`，header 打印出来。
- 这样你拧动 drama 后能直接在 trace 顶部看到当前档位。

**验收标准**：
- `python watch.py 20 --reset` 跑两次：先把 `world_state.drama` 设为 `0`，再设为 `80`（直接改库或加个临时设置即可）。
- 对比两次统计：`drama=80` 时"有事件的拍 %"应明显高于 `drama=0`，且触发的事件 title 应更多落在"针对某个缺钱/低落角色"的那几个新事件上。
- `drama=0` 时的行为应与阶段 1 改动前**基本一致**（回归保护）。
- 全程不得绕过 `executor.py` 的 allowlist；不得扩展事件 condition 的 DSL。

## 阶段 2：把目标接进闭环（**故意推迟，先别做**）

> 这一步是让"模拟器"真正变成"故事/自动电影"的关键，但它更重，且**只有在你用阶段 0 的观察窗看过几十拍真实运行之后，才有足够依据决定怎么做**。现在硬上等于又在凭空猜规格。
>
> 推迟它本身是方法的一部分：先有 walking skeleton（阶段 0+1），看见真实行为，再决定阶段 2。

阶段 2 的方向（届时再细化成独立文档）：
- 让 `goals` 进入**裁决**提示词，并允许裁决识别"本拍是否推进/挫败了某角色的某条目标"。
- 可能需要一个轻量的"目标进度"表示（不一定是新数值字段，先用 scene_log 里的叙述标记起步）。
- 复用 executor 里已存在但被 judge schema 挡住的 `world_state` delta 能力（`tension`/`economy_index`），让"导演"能调氛围。
