# 设计三:提示词与模型交互设计

> 数据库见 `01_数据库设计.md`,引擎见 `02_引擎程序设计.md`。
> 后端:LM Studio 单模型(建议 Qwen3.6 30B 级、tool/JSON 输出可靠的)。**好不好玩的真正瓶颈在这一份文档。**

## 0. 三个角色,一个模型

同一个 weights,用不同 system prompt 切三种身份。三者契约严格分工:

| 身份 | 职责 | 绝不做 | 输出 |
|---|---|---|---|
| **角色(Character)** | 以 persona 产出一句话/一个行动 | 不裁定后果、不碰数值 | `ACTION` JSON |
| **裁定(Judge / GM)** | 判成败、给叙事、给状态**变化量** | 不给绝对值、不写库 | `VERDICT` JSON |
| **世界(World)** | 把触发的事件演绎成具体情节 | 不替角色说话、不凭空造人 | `EVENT` JSON |

> 三者都**只输出 JSON**,由引擎解析校验。任何越权(角色给 delta、裁定写绝对值、世界造注册表外的人)在解析层就被丢弃。

---

## 1. 通用约定(三者共享)

- **严格 JSON 输出**:system prompt 末尾固定一句"只输出 JSON,不要任何解释、不要 markdown 代码块"。LM Studio 开 **JSON mode / structured output**(若模型支持 grammar 约束更稳)。
- **关思考冗余**:Qwen 系列爱过度 reasoning。长跑里要么关 thinking,要么限制思考长度 —— 否则速度和 token 双爆。
- **前缀稳定 = KV cache 友好**:把"写死、不变"的内容(persona 内核、世界观、规则)放在 prompt **最前面且固定**,把"每拍变化"的放后面。前缀不变,LM Studio 能复用 KV cache,prefill 提速明显。
- **数值全部来自 DB 实时注入**,模型从不自己记账。

## 2. 角色层 Prompt

### 2.1 三层上下文组装(对应引擎 §3)

```
[System] = 角色内核(写死,最前,稳定) + 角色当前状态(DB实时) + 输出契约
[System 续] = 舞台(当前场景:地点/在场者/世界事件)
[Messages] = 本场景最近若干拍(经可见性过滤的对话历史)
```

### 2.2 System 模板

```
你正在扮演【{name}】。严格以这个人的性格、价值观、说话风格行动。

# 你是谁(这是你的本性,不会改变)
背景:{backstory}
性格(0-100):{personality}     # 高自制力=克制,高攻击性=易冲突...
价值观:{values}
说话风格:{speech_style}        # 必须贯穿你的每句话
雷点:{taboos}                   # 触及时按你的性格强烈反应

# 你现在的状态(实时,会变)
心情{mood} 精力{energy} 醉意{intoxication} 钱{money} 名声{reputation}
你记得的人:                      # 只注入在场者的关系记忆
  - {present_char}: 好感{aff} 信任{trust} | {memory_text}
你最近的记忆:{salient_memories}   # 高salience摘要 + 最近episodic
你当前的目标:{current_goals}

# 你能做什么
- 说话;或做一个动作(action)。动作必须符合你的能力与在场环境。
- 只能与"在场的人"和"存在的地点/物品"互动,不要提及不存在的人或物。
- 如果想离开,用 exit 动作。

# 输出(只输出 JSON)
{ACTION_SCHEMA}
```

### 2.3 ACTION 契约

```json
{
  "speech": "（可空）你说出口的话,贴合 speech_style",
  "inner": "（可空，≤15字）极简心理活动,用于裁定参考,不展示给其他角色",
  "action": {                       // 可空:没有行动就只 speech
    "type": "buy | give | attack | work | move | exit | custom",
    "target": "char_id 或 item 或 location_id 或 null",
    "detail": "自然语言描述意图,如『赊一杯烈酒』"
  },
  "to": "char_id 或 null"            // 这句话主要对谁说(供引擎接力调度)
}
```

> `to` 字段让引擎零成本接力选下一个发言者(引擎 §9)。`inner` 给裁定层一点内心依据,但极短以省 token。
> 当前 MVP 每拍都会进入裁定层，但普通 speech 应返回空 `deltas`；这样可以让明确针对某人的一句话产生受控的角色/关系影响，同时保持结果可见、可回放。

## 3. 裁定层 Prompt

### 3.1 System 模板

```
你是这个世界的规则裁定者(GM)。给定一个角色的行动,判断其结果,并给出对世界状态的**变化量**。

# 裁定原则
- 依据行动者的能力、性格、当前状态,以及在场环境,判断成败与后果,要合理、有因果。
- 同一行动,不同能力/性格/醉意的人结果不同(自制力低+醉意高→更易失控)。
- 你只给【变化量 delta】,绝不给绝对数值。不要假设任何人有多少钱/好感,以输入为准。
- 后果要克制:当前 MVP 单项变化通常在 ±5 内；普通寒暄不应产生 delta。
- narration 用第三人称客观叙述结果,简洁,不替角色说台词。

# 输入
行动者状态:{actor_state}
行动:{proposal}
在场者:{present_with_rel}
世界:{world_brief}

# 输出(只输出 JSON)
{VERDICT_SCHEMA}
```

### 3.2 VERDICT 契约(引擎 executor 据此落账)

```json
{
  "success": true,
  "narration": "酒保皱眉,还是把酒推了过来。",
  "deltas": [
    {"target":"character_attribute","ref":"actor_id","field":"money","op":"add","value":-5,"reason":"付出酒钱"},
    {"target":"character_attribute","ref":"actor_id","field":"mood","op":"add","value":1,"reason":"得到回应"},
    {"target":"relationship","source":"actor_id","ref":"bartender_id","field":"affection","op":"add","value":-2,
     "reason":"公开赊账被对方记下"},
    {"target":"skill","ref":"self","field":"xp","skill":"酒量","value":3}
  ],
  "triggers": ["check_drunk"]            // 可空:提示引擎检查某涌现事件
}
```

> `field` 必须落在**字段白名单**(数据库 §6),否则 executor 丢弃。`value` 是增量,executor 负责夹 0-100。关系变化只能从当前 actor 指向本拍 `to` 或 action target 所指的在场角色；本地预检会拒绝其余关系 delta。

## 4. 世界层 Prompt

### 4.1 System 模板

```
你是这个世界的导演。一个事件被触发了,请把它演绎成具体、贴合世界观的情节。

# 世界观(根基,不变)
{worldview}

# 当前世界
时间{period} 天气{weather} 紧张度{tension} 进行中的剧情线:{active_threads}
触发的事件:{event_guidance}        # 来自 event_defs.guidance
相关角色:{involved_chars}           # 只能涉及这些已存在的角色

# 原则
- 只用已存在的角色、地点;绝不创造新人物或新地名。
- 事件要呼应当前世界状态与剧情线,推动而非打断。
- 给出事件叙述,以及它对世界/在场者的客观影响(delta)。
- 控制规模:一次事件别同时改太多东西。

# 输出(只输出 JSON)
{EVENT_SCHEMA}
```

### 4.2 EVENT 契约

```json
{
  "narration": "来生酒吧的全息门帘闪了一下,两个催收员扫视主厅,目光锁定角落的{char}。",
  "location_id": "main_floor",
  "visibility": ["all"],             // 或指定 char_id 列表(私密事件)
  "deltas": [
    {"target":"world","field":"tension","op":"add","value":20},
    {"target":"char_state","ref":"char_debtor_target","field":"mood","op":"add","value":-25}
  ],
  "thread": {"id":"debt_collection","desc":"债主追讨","stage":"escalate"}  // 可空:更新剧情线
}
```

> 事件的 narration 作为"环境消息"注入 scene_log,在场角色下一拍上下文带上它 → 影响言行。`thread` 让长剧情跨场景延续(写回 world_state.active_threads)。

## 5. Response Filtering(防 OOC / 防幻觉)—— 长跑质量保障

角色输出落库前过两道(引擎 §7),对应 Agentopia 的 roleplay principles,精简到本项目够用的几条:

**代码层(硬过滤,不调 LLM):**
1. 提到的 char_id / location 不在注册表 → 驳回重生成。
2. JSON 非法 / 越权字段 → 重生成或丢弃。
3. 与上一拍近乎复读(相似度过高)→ 重生成。

**LLM 层(裁定/世界模型顺带判,或独立轻量调用):**
4. 是否严重 OOC(违背 personality / speech_style)。
5. 是否违反角色 taboos 设定的反应方向。
6. 是否擅自替别人决定行动/说话。

违规则要求角色模型重生成一次(prompt 末尾追加具体违规点);二次仍违规则降级为"沉默/无动作",保证不卡死。

## 6. 模型与采样配置(LM Studio 30B)

| 身份 | temperature | 说明 |
|---|---|---|
| 角色 | 0.8~0.95 | 要个性与变化,别太死板;配 frequency_penalty 0.05~0.1 压复读/AI腔 |
| 裁定 | 0.3~0.5 | 要稳定、可复现的判决,低温 |
| 世界 | 0.7 | 要创意但受控 |

通用:
- 开 **JSON / structured output**;能上 **GBNF grammar** 约束就上,JSON 合法率显著提升。
- `max_tokens` 收紧(角色 ~256、裁定 ~512、世界 ~512),长跑省时省 token。
- **关闭或限制 reasoning**;若用 thinking 模型,把 think 段从落库内容里剥掉,只留最终 JSON。
- 上下文长度按 30B 实际可用设;**靠记忆分级检索控制输入量**,不要堆满 context(输入主导成本,输入越长每拍越慢)。

## 7. 一拍完整交互时序(把三份文档串起来)

```
clock.tick                                   ── 代码
eventsys 判定 → (命中) 世界模型生成 EVENT ──── LLM#世界(偶发)
  → executor 落 world/char deltas,注入 scene_log
scheduler 选 actor(接力/导演)                ── 代码 (+偶发LLM#导演)
agent.act → ACTION                           ── LLM#角色 (每拍必有)
  代码: response filter(实体/复读)
proposal.has_effect?
  是 → judge.adjudicate
        precheck(代码) → VERDICT             ── LLM#裁定 (有后果才有)
        executor.apply(校验+夹边界+写库)      ── 代码
        growth.accrue(技能xp/关系)            ── 代码
  否 → 直接落 scene_log,跳过裁定             ── 省一次调用
memory.record / 周期摘要 / 章末成长结算       ── 代码 (+章末LLM#世界做人格漂移)
append scene_log → 回放端异步消费
```

**每拍 LLM 调用数:** 日常闲聊 = 1(角色);有行动 = 2(角色+裁定);触发事件 = 3。把闲聊保持在 1,是这套系统能在 30B 上"一直跑"的根本。

---

## 8. 落地次序建议

1. 先只做**角色 + 裁定**两个 prompt,手动塞一个固定场景,验证"提议→裁定→落库"闭环跑通、JSON 稳。
2. 加 response filtering,确认长跑不漂、不复读。
3. 加世界层与事件,世界开始自走。
4. 加章末成长结算,观察 8~12 个角色跑几章后,关系网与人格是否出现有意义的演化。

到第 4 步,你就有了一个"角色有成长、世界有变化、能自己一直跑"的最小可行 AI 角色世界。
