# 设计四:上下文无关游戏引擎设计

> Agentopia 的核心不是"酒馆聊天",而是一个可替换题材的 AI 游戏引擎。
> 数据库见 `01_数据库设计.md`,引擎见 `02_引擎程序设计.md`,提示词契约见 `03_提示词与模型交互设计.md`。

## 0. 核心判断

Agentopia 必须被设计成**上下文无关的游戏引擎**。

这里的"上下文无关"不是说 LLM 没有上下文,而是说:

- 不依赖把完整世界观、完整角色卡、完整历史、完整关系网全塞进 prompt。
- 不让 prompt 随运行时间无限膨胀。
- 每次 LLM 调用只接收**固定系统契约 + 当前数据库切片 + 本拍任务**。
- 角色和世界的变化先进入数据库,下一拍再从数据库重新取新状态组装上下文。

所以长跑稳定性的第一原则不是"上下文足够大",而是**每拍重新从真相源取当前状态**。

## 1. Agentopia 首先是游戏引擎

Agentopia 的题材可以是酒馆、城镇、学院、飞船、荒野营地,但底层能力应该一致:

```
当前状态(SQLite) → AI 生成语义与意图 → 引擎校验/裁定/落库 → 新状态(SQLite)
```

LLM 在这里承担的是"活的游戏逻辑层":

- 根据当前状态生成角色意图、语言、行动。
- 根据行动和规则生成结果、变化量、叙事。
- 根据世界状态生成事件推进。

但数据库写入仍必须走引擎边界:

- LLM 可以提出 delta。
- executor 负责白名单、边界夹紧、合法性检查和写库。
- 最终状态只以 SQLite 为准。

换句话说:AI 是游戏引擎的语义处理器,SQLite 是游戏引擎的内存和存档,executor 是写入总线。

## 2. 不做"大上下文角色扮演"

传统做法容易变成:

```
完整世界观 + 完整角色卡 + 完整历史 + 完整关系 + 最近对话 → LLM
```

这个路线的问题是:

1. prompt 会越来越大。
2. 压缩会导致细节变形。
3. 长历史会污染当前判断。
4. 题材一复杂,每拍成本失控。
5. 角色变化后,旧上下文里的角色印象会和新状态冲突。

Agentopia 必须避免这条路。

## 3. 每拍上下文结构

每次 LLM 调用只允许组装必要切片:

```
[固定系统提示]
  - 输出 JSON 契约
  - 引擎规则
  - 当前 LLM 身份(Character/Judge/World)

[当前数据切片]
  - 当前角色的核心属性和动态状态
  - 当前场景里在场角色的必要摘要
  - 与本拍相关的关系/记忆/地点/事件
  - 场景对话状态(当前话题、上一位发言者/对象、最后一句、公开听众)
  - 最近少量结构化 scene_log(谁说、对谁说、谁听到、行动目标、话题)

[本拍任务]
  - 你现在要说话/行动
  - 或你现在要裁定一个 action
  - 或你现在要演绎一个事件
```

禁止把"所有角色、所有记忆、所有历史、所有世界设定"塞进单次调用。

## 4. 数据变化驱动上下文变化

角色和世界一直在变,所以 prompt 里的数据也必须来自最新数据库。

例:

```
tick 10: 林澈 money=12 mood=50
LLM 提出买酒 → Judge 给 money -3, mood +4
executor 写库
tick 11: 下一次 prompt 读取到 money=9 mood=54
```

LLM 不需要记得"刚才买酒后应该少钱";它只需要看到数据库里的当前状态。

这就是上下文无关的关键:

- 历史不靠模型记。
- 状态不靠模型记。
- 当前事实从 DB 注入。
- 模型只处理"此刻该怎么反应"。

## 5. 固定背景提示与可变数据分离

prompt 必须分成两类:

### 固定部分

- 引擎规则
- 输出 schema
- 世界类型的根规则
- 角色/裁定/世界三种身份契约

这些内容应该尽量稳定,放在 prompt 前缀,方便 KV cache。

### 可变部分

- 当前 tick/day/chapter
- 当前角色状态
- 当前地点状态
- 在场角色
- 最近 scene_log
- 当前场景的一行 conversation state
- 检索出的少量相关记忆
- 本拍任务

这些内容每拍从 SQLite 读取,不能靠内存猜测,也不能靠上一轮 prompt 延续。

## 6. 对实现的要求

后续所有模块都要遵守:

1. `agent.py`/`engine.py` 不组装全量世界,只组装当前 actor 所需切片。
2. `judge.py` 只接收待裁定 action 和必要状态,不读全量历史。
3. `memory.py` 的职责是检索相关记忆,不是把历史全塞回去。
4. `eventsys.py` 只把命中的事件上下文给世界模型,不让世界模型自由遍历全世界。
5. `executor.py` 是唯一写库入口,但它服务的是 AI 驱动的游戏逻辑。
6. 前端只是观察和干预,不能成为状态真相源。

## 7. 判断一个功能是否跑偏

如果一个功能需要:

- 增加大量 prompt 长文才能维持角色一致性;
- 每拍塞入越来越多历史;
- 依赖模型记住之前发生过什么;
- 角色数据在 prompt 和 DB 中出现两个版本;
- 为了某个题材写死大量特殊逻辑;

那它就在偏离 Agentopia。

正确方向是:

- 把事实落库。
- 把变化表达成 delta。
- 把题材差异做成数据定义。
- 每拍只取当前需要的切片。
- 让 AI 处理语义,让引擎处理边界和持久化。

## 8. 一句话原则

**Agentopia 不是靠更长上下文模拟世界,而是靠数据库状态循环驱动世界。**
